## Context

The NetPractice scraper (`scrapers/netpractice.py`) currently connects to Chrome via CDP on port 9222, but Chrome 136+ (May 2025) broke the old `--remote-debugging-port` approach for default profiles. Chrome now requires a dedicated `--user-data-dir` to accept the debug flag. None of our launch scripts or scrapers handle this. Additionally, users need the ability to manually add patients to their schedule (walk-ins, add-ons, personal prep lists) and trigger Amazing Charts data collection from the dashboard with a clear mouse/keyboard takeover warning.

### What Already Exists
- `scrapers/netpractice.py` — 1,300+ line async Playwright scraper with DOM-first parsing, nav step replay, date navigation, cookie persistence to `data/np_session.pkl`
- `scrapers/pdmp.py` / `scrapers/viis.py` — Same CDP-first + headless fallback pattern
- `models/schedule.py` — `Schedule` model with `entered_by` field (already supports distinguishing manual entries)
- `routes/dashboard.py` — Today View with schedule table, anomaly detection, billing opps, duration estimator
- `templates/dashboard.html` — Full dashboard with widgets, date navigation, patient panel, XML drop zone
- `routes/netpractice_admin.py` — Admin settings + setup wizard + nav step recorder
- `agent/clinical_summary_parser.py` — `open_patient_chart(mrn)` + `export_clinical_summary()` for AC automation
- `agent/ac_window.py` — `get_ac_state()`, `get_current_patient()` for AC detection
- `agent/pyautogui_runner.py` — Safety-checked PyAutoGUI execution with pre/post screenshots
- `config.py` — `CHROME_CDP_PORT = 9222`, NetPractice URL/client number, session cookie path

### Key Chrome 136 Facts
- `--remote-debugging-port` is **ignored** when Chrome uses the default user data directory (Chrome 136+, May 2025)
- Must launch with BOTH `--remote-debugging-port=9222` AND `--user-data-dir=<custom-path>`
- `--remote-allow-origins=*` required since Chrome 111 (WebSocket origin check)
- Cookies persist in the custom profile directory — log in once, reuse indefinitely
- `connect_over_cdp()` is the correct Playwright method (not `connect()`)

## Architecture Decisions

1. **Playwright stays** — all 3 scrapers already use it; no reason to switch to Selenium or raw CDP
2. **Single Chrome debug profile per machine** — one `chrome-debug-profile` directory, matching the single-provider deployment model
3. **Manual entries use `entered_by='manual'`** — coexist with scraped entries, visually flagged, deletable
4. **Note template stored in `Schedule.comment`** — avoids DB migration; format `"template:SOAP"` prefix. Parsed back out in UI when needed.
5. **Data collection reuses existing AC pipeline** — `clinical_summary_parser.py` already handles open chart + export + parse. No new PyAutoGUI code.
6. **Session expiry: re-auth on demand** — debug profile persists cookies. If session expires, scrapers detect login redirect and user re-authenticates manually. Existing re-auth banner already handles this.
7. **Parse-text is local regex only** — no API calls, no AI. Fast, deterministic, works offline. Users can always edit results before committing.
8. **Background thread for data collection** — not celery/rq, just `threading.Thread`. Simple, sufficient for single-user desktop app. Job state in module-level dict (lost on server restart, which is fine — collection takes <60 seconds).

9. **P1 15-second poll** is limited to `priority=1 AND acknowledged_at IS NULL` — lightweight query with DB index
10. **Empty state and WhyLink macros are Jinja2 includes** — zero API overhead
11. **Widget layout persisted both client-side (localStorage) and server-side (UserPreference)** — fast load + cross-session persistence

> **Prerequisite**: Running Plans 1–2 complete, Part 2 Phases 6–8 complete (API bug fixes, widget grid overhaul)

## Architecture Decision — Three-Tier Waterfall Rationale

This feature was originally designed around GoodRx as the primary pricing source. Revised based on three findings:

1. **GoodRx Coupon API minimum**: Requires 3,000 coupon accesses/month. A single-provider practice with 20–25 patients/day cannot meet this. Coupon API excluded.
2. **GoodRx aggregation prohibition**: ToS prohibits displaying GoodRx prices alongside competing pricing sources in the same UI component.
3. **Cost Plus Drugs free public API**: No key, no auth, no rate limits, no minimums, no ToS conflicts.

**Revised architecture**: Cost Plus Drugs is the primary source (Tier 1). GoodRx Price Compare (cash prices only, no coupons) is the secondary source (Tier 2) — called only when Cost Plus returns no result, ensuring GoodRx and Cost Plus prices never appear in the same card. NeedyMeds/RxAssist (Tier 3) provide patient assistance programs for expensive medications or uninsured/Medicaid patients.

## SECTION 1 — AUDIT OF EXISTING STRUCTURE

### Audit Question 1: RxNorm Normalization Pipeline

**Location**: [app/services/api/rxnorm.py](app/services/api/rxnorm.py) — `RxNormService` class inheriting from `BaseAPIClient`

**What it returns**: The pipeline has two levels:
- `get_rxcui(drug_name)` → `{rxcui, name, _stale}` — resolves any drug string to an RxCUI
- `get_properties(rxcui)` → `{name, synonym, tty, language, suppress, umlscui, _stale}`
- `get_ingredient(rxcui)` → `{ingredient_rxcui, ingredient_name, _stale}` — strips dose form/strength
- `get_ndcs(rxcui)` → `{ndcs: [list of NDC strings], _stale}` — **NDC retrieval already exists**

**NDC status**: ✅ The `get_ndcs()` method at [rxnorm.py line 127](app/services/api/rxnorm.py#L127) already calls `/rxcui/{rxcui}/ndcs.json` and returns a list of NDC codes. It uses `RXNORM_NDC_TTL_DAYS = 7` from api_config.py. **No additional RxNorm endpoint call is needed.** The Cost Plus API can receive the first NDC from this list directly.

**Patient chart enrichment path**: [routes/patient.py](routes/patient.py) function `_enrich_rxnorm()` (line ~415) resolves RXCUI → RxNormCache row. The `RxNormCache` model ([models/patient.py line 233](models/patient.py#L233)) stores brand_name, generic_name, dose_strength, dose_form, route, tty — but does **not** currently store NDC. The NDC is available via `RxNormService.get_ndcs()` but is not persisted in the cache row.

**Gap to fill**: Add an `ndc` column to `RxNormCache` and populate it during enrichment so the pricing waterfall can access NDC without an extra API call at display time.

### Audit Question 2: Medication Reference Module

**Location**: [routes/medref.py](routes/medref.py) — `medref_bp` blueprint, template at [templates/medref.html](templates/medref.html)

**Current data sources displayed**:

- OpenFDA Drug Label: indications, dosage, boxed warnings, warnings, contraindications, adverse reactions, drug interactions, pregnancy/lactation, renal/hepatic
- OpenFDA FAERS: top 5 real-world adverse events with counts
- OpenFDA Recalls: active recall status with classification

**Layout**: Three-column grid — left panel (search + quick filters + history, 360px) and middle panel (drug card). ### Audit Question 5: api_config.py Pattern

**Location**: [app/api_config.py](app/api_config.py)

**Free API pattern**: The file is organized into tiered sections (`TIER 1`, `TIER 2`, `TIER 3`, `TIER 4`) with clearly labeled comment blocks. Free APIs with no authentication use the pattern:
```python
# Description comment — purpose (no auth)
API_NAME_BASE_URL = "https://..."
API_NAME_CACHE_TTL_DAYS = N
```
No `.env` entry, no config.py import, no runtime key check. Examples: `RXNORM_BASE_URL`, `ICD10_BASE_URL`, `NLM_CONDITIONS_BASE_URL`, `OPEN_METEO_BASE_URL`.

APIs with keys use a `try/except` block at the bottom that imports from `config.py` (lines 117–126): `OPENFDA_API_KEY`, `PUBMED_API_KEY`, `UMLS_API_KEY`, etc.

### Audit Question 6: Unified Cache Manager

**Location**: [app/services/api/cache_manager.py](app/services/api/cache_manager.py) — `CacheManager` class

**Pattern for registering a new source**: Registration is implicit. Any `BaseAPIClient` subclass passes `api_name` and `ttl_days` to `super().__init__()`. The `CacheManager` stores entries in a single `api_response_cache` table keyed by `SHA-256(api_name::query_key)`. The `set()` method takes `api_name`, `query_key`, `data`, and `ttl_days`.

**Registering Cost Plus (72-hour TTL)**: `super().__init__(api_name="cost_plus", base_url=COST_PLUS_BASE_URL, db=db, ttl_days=3)` — the TTL is 3 days = 72 hours. Individual calls can override via `self._get(path, ttl_days=3)`.

**Registering GoodRx (24-hour TTL)**: `super().__init__(api_name="goodrx", ..., ttl_days=1)`.

**NeedyMeds/RxAssist (7-day TTL)**: `ttl_days=7`.

### Audit Question 7: Base Client Authentication Support

**Location**: [app/services/api/base_client.py](app/services/api/base_client.py) — `BaseAPIClient` class

**Current behavior**: The `_get()` method builds a `urllib.request.Request` with only `User-Agent` and `Accept: application/json` headers. There is no authentication configuration in the base class — no API key parameter, no auth header injection, no signing logic.

**Unauthenticated GET support**: ✅ The base class already supports unauthenticated GET requests with query parameters perfectly. `_get(path, params={...})` builds a URL with `urlencode(params)` and makes a standard GET. Cost Plus requires exactly this — no changes to the base class needed.

**HMAC-SHA256 for GoodRx**: The base class does **not** support HMAC signing. The GoodRx Price Compare API requires signing each request with HMAC-SHA256 using the secret key. The GoodRx service module will need to override `_get()` or implement a custom `_signed_get()` method that: (1) builds the query params, (2) computes the HMAC signature, (3) appends the signature to the params, (4) calls the existing `_get()` logic. This can be done entirely within the GoodRx subclass without modifying the base class.

## Part 3 Architecture Decisions

1. **Cost Plus is Tier 1 (primary)** — free, no key, no ToS conflicts, covers majority of primary care generics. Always queried first.
2. **GoodRx is Tier 2 only** — Price Compare endpoint only, Coupon API intentionally excluded (3,000/month minimum). Only queried when Cost Plus returns no result. ToS no-aggregation constraint satisfied by waterfall mutual exclusivity.
3. **NDC is the preferred lookup key** for Cost Plus (most precise match). The RxNorm pipeline already provides `get_ndcs()` — we just need to persist the first NDC in `RxNormCache`.
4. **Pricing is non-critical** — all service calls wrapped in try/except. A pricing API failure never causes the application to error. Badge simply not shown, no error displayed.
5. **GoodRx HMAC signing stays in the subclass** — no modification to `BaseAPIClient` needed. The GoodRx service overrides with a custom `_signed_get()` method.
6. **Cache TTLs are source-appropriate**: Cost Plus 72h (prices update periodically), GoodRx 24h (prices change more frequently), Assistance programs 7d (programs change infrequently).
7. **Morning briefing pricing refresh is scoped to today's patients only** — limits API calls and respects GoodRx's abuse monitoring. Not a full-panel refresh.
8. **Demo mode covers all 3 tiers** — seeded cache data enables full UI testing without any API keys. Cost Plus can go live immediately without waiting for GoodRx approval.
9. **NADAC (pharmacy acquisition cost)** identified as a future enhancement — free CMS data that could add a "pharmacy cost" baseline. Deferred to avoid scope creep.
10. **Patient education pricing language is informational only** — never suggests the practice endorses any pricing service. Uses neutral phrasing per HIPAA/compliance guidance.
